# Phase 15.1 Research: P0 Basic Editing Chain Repair

**Date:** 2026-06-19
**Status:** Ready for planning

## Executive Findings

Phase 15.1 is not a visual polish task. The current editor can show a first frame, but normal playback is still not a production realtime playback chain:

- `apps/desktop-electron/src/renderer/App.tsx` advances the playhead with browser `requestAnimationFrame` and repeatedly calls `requestPreviewFrameAt`.
- `PreviewMonitor.tsx` and `Timeline.tsx` disable play when `runtimeDiagnostics.canPreview` is false, but "can preview" still primarily means the FFmpeg/artifact preview service is available.
- `apps/desktop-electron/src/main/realtimePreviewHost.ts` attaches a native/offscreen preview host and polls telemetry, but it only has an internal test frame path and no product playback control surface for play/pause/seek/scrub.
- `crates/realtime_preview_runtime` already owns `TimelineClock`, `PlaybackGeneration`, `PlaybackState`, session seek, and frame request telemetry. The missing layer is binding/main/preload/renderer integration for playback controls and draft snapshot handoff.
- `crates/bindings_node/src/realtime_preview_service.rs` already has registry methods for create, close, attach, update draft snapshot, seek, request frame, cancellation, and telemetry. It does not yet expose play/pause/stop/resume through the Node binding and desktop host API.

The user-facing symptom follows from this mismatch: the first frame can be produced, but play is either disabled by runtime capability checks or runs a UI-owned still-frame loop rather than a realtime preview runtime.

## Codebase Facts

### Playback and preview

- `App.tsx` imports and uses `buildRequestPreviewFrameCommand` / `buildRequestPreviewSegmentCommand`.
- `App.tsx` playback loop computes elapsed time in React and calls `requestPreviewFrameAt(targetTime)`.
- `PreviewMonitor.tsx` sends host bounds through `window.videoEditorRealtimePreviewHost.updateHostRect`, then renders realtime host status and fallback overlays inside the canvas.
- `realtimePreviewHost.ts` creates a realtime preview session, attaches/updates the surface, and records telemetry, but it never receives the current draft snapshot from renderer state and never exposes play/pause/stop commands.
- `realtime_preview_runtime::TimelineClock` has `play`, `pause`, `resume`, `stop`, `seek`, and generation handling.
- `RealtimePreviewRuntime` exposes `seek`, `pause`, `update_draft_snapshot`, and `request_frame`; it needs public play/stop/resume equivalents and binding coverage.
- Existing tests in `crates/realtime_preview_runtime/tests/clock_generation.rs`, `stale_frame_rejection.rs`, and `cancellation_telemetry.rs` are the closest patterns.

### Font registry and text parity

- `media_runtime/src/capabilities.rs` probes `VE_TEXT_FONT_PATH` and some host font paths, but this is environment discovery, not a bundled product registry.
- `realtime_preview_runtime/tests/text_parity.rs` explicitly treats GPU text parity as unsupported without proven repository fonts.
- `artifact_store/tests/resource_index.rs` has resource index patterns for font resource refs.
- Text panel defaults still create `fontRef: null`; export/preview can therefore diverge through host fallback fonts.
- Phase 15.1 needs one bundled, license-tracked, Chinese-capable font entry with stable `fontRef`, file path, family/style metadata, and validation. The exact font can be selected during implementation, but it must be open-source and redistributed with license metadata.

### First-material canvas adaptation

- `draft_commands/src/canvas.rs` only has explicit `update_draft_canvas_config`.
- Importing material and adding a segment do not currently carry a Rust-owned "canvas was manually changed" policy.
- `commandHelpers.ts` can build `updateDraftCanvasConfig`, but renderer-owned canvas inference would violate Phase 15.1.
- The right owner is a Rust command/service helper that receives accepted draft + material metadata and returns an atomic timeline response or material import response with canvas update events.
- Default segment visual currently includes fit modes; Phase 15.1 should ensure first visual placement avoids stretch distortion for portrait/square media.

### Multitrack editing

- `draft_model::Track` supports `track_id`, `kind`, `name`, `muted`, `locked`, and `segments`.
- `draft_commands::timeline` has stack/mix order helpers and set mute, but no user-facing add track, rename track, set track lock, or set track visibility command.
- Draft model has `muted` and `locked`; it lacks a persisted visual track visibility flag. Phase 15.1 can add a backward-compatible `visible` field to `Track` or model visibility through visual semantics, but UI track show/hide needs a Rust-owned command.
- `Timeline.tsx` renders lock/visibility/mute controls, but lock and visibility are disabled placeholders.

### Save, reopen, export user chain

- Real workflow tests exist in `apps/desktop-electron/tests/real-workflow.spec.ts` and helper `tests/helpers/realWorkflow.ts`.
- Project persistence tests exist under `crates/project_store/tests/project_bundle.rs`.
- Existing E2E focuses import-preview-export. Phase 15.1 needs a stricter flow: vertical video import, auto canvas, playback without repeated `requestPreviewFrame`, bundled-font text, music, trim, split, undo/redo, save, reopen, export.
- `recordTestExecuteCommand` in `apps/desktop-electron/src/main/index.ts` already records `requestPreviewFrame` and can be extended to record realtime preview host control calls for source-level assertions.

## Recommended Plan Shape

Use six plans:

1. Playback chain repair: add failing tests proving play does not call `requestPreviewFrame` repeatedly; expose realtime play/pause/stop/seek draft snapshot APIs; wire desktop UI play controls to host/runtime.
2. Baseline preview capability gate: classify supported baseline video/image/text/audio preview and productize unsupported failures instead of accepting artifact fallback as normal playback.
3. Bundled font registry: add font assets/manifest/registry, generated contracts, UI default `fontRef`, and export/preview validation.
4. First-material canvas adaptation: Rust-owned canvas adoption helper/command path, default fit behavior, and tests for portrait/square/landscape plus manual override.
5. Multitrack commands and UI: add/rename/lock/show/mute/target track commands with generated contracts, undo/redo, deterministic stacking/mix tests, and minimal UI controls.
6. Full user-chain acceptance: real E2E plus source guards proving default playback is realtime, debug preview commands are not the normal path, save/reopen/export passes, and Phase 16 remains blocked until this gate passes.

## Risks

- A cosmetic play-button enablement would hide the root issue. The tests must count command calls and realtime host calls.
- Adding font assets has licensing risk. Implementation must include license metadata and not rely on proprietary host fonts.
- Canvas adoption can corrupt user intent if it runs after manual canvas edits. The plan must persist or infer a reliable "manual canvas" state.
- Multitrack visibility requires a schema addition. It must be backward-compatible and generated to TypeScript.
- Full no-mock E2E may be slower; keep it as an explicit phase gate rather than running inside every narrow unit test.

## Research Complete

The planning surface is sufficiently understood. Implementation should begin with playback-chain RED tests because they protect the user-visible P0 bug: first frame appears but play is disabled or fake.
