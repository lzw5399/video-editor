## User Constraints (from CONTEXT.md)

### Locked Decisions
## Implementation Decisions

### Audio Runtime Ownership
- **D-01:** Audio preview playback is Rust-owned. Electron/React may issue play, pause, seek, cancel, and device-selection commands, but it must not construct audio graphs, mix buffers, FFmpeg commands, output-device handles, cache keys, or timeline state.
- **D-02:** The audio runtime must use the existing `TimelineClock` and `PlaybackGeneration` concepts from `realtime_preview_runtime`. Seek, pause, resume, accepted edits, draft reloads, and material relinks must advance/reuse generation rules so stale audio buffers cannot overwrite current playback.
- **D-03:** Preview audio scheduling and buffering must be independent from FFmpeg preview frame generation. FFmpeg may remain an export/fallback/runtime source where existing crates require it, but supported interactive audio preview cannot depend on per-frame/per-segment FFmpeg preview requests.

### DSP Timeline Semantics
- **D-04:** Segment gain, track mute, pan, fades, and keyframed volume must be represented as typed Rust DSP timeline semantics using integer microseconds, frame/sample indices, rational rates, and bounded integer gain units. Do not persist naked floating-point seconds in draft semantics.
- **D-05:** Future audio effects need extension points in the DSP graph, but Phase 15 should only implement the semantic carriers and classifier needed to keep later effects from changing ownership boundaries.
- **D-06:** Draft commands remain the source of accepted audio semantic edits. The audio graph consumes accepted draft state and command deltas; it does not mutate draft state or invent timeline edits.

### Preview, Export, And Parity
- **D-07:** Preview and export audio mixdown must share the same Rust DSP evaluation path wherever possible. Any unavoidable preview/export differences must be classified in parity diagnostics rather than hidden in UI behavior.
- **D-08:** Export audio remains integrated with the existing render/export pipeline, but `ffmpeg_compiler` should consume audio mix intent generated by Rust semantics. The renderer must not assemble FFmpeg filter graphs or arguments for audio.
- **D-09:** Audio parity tests should use deterministic fixtures from `testkit` and compare typed mix intent/sample summaries before relying on heavyweight media binaries.

### Platform Output Boundary
- **D-10:** macOS and Windows output support should be represented behind Rust traits and capability reports. macOS targets CoreAudio and Windows targets WASAPI, but Phase 15 may introduce mock/stub device implementations where direct OS playback is not yet feasible in CI.
- **D-11:** Native audio output handles and buffers must not cross into TypeScript. Bindings expose opaque session IDs, command payloads, safe status summaries, device capability summaries, and diagnostics.

### Waveform And Artifact Store Boundary
- **D-12:** Waveform and peak data from `artifact_store` are derived display artifacts. They can drive timeline UI rendering, but they must not become canonical audio semantics in `.veproj/project.json` or the DSP evaluator.
- **D-13:** The desktop waveform UI should consume safe artifact summaries and display-ready peak payloads through generated command/binding contracts. It must not read SQLite, blob paths, fingerprints, dirty ranges, or artifact roots directly.

### Desktop Product Boundary
- **D-14:** Phase 15 should remove or avoid debug-heavy audio UI. Production-facing desktop UI should show only playback state, waveform availability, safe buffering/status labels, and controls needed for Jianying-style editing.
- **D-15:** The UI direction remains Jianying-style with feature reduction: familiar playback/seek/timeline behavior and compact controls, but no unsupported digital-human or unrelated advanced panels.

### the agent's Discretion
The planner may choose exact crate names and module splits for the audio graph, DSP evaluator, output traits, bindings, and tests. It should prefer small Rust crates with clear ownership, source guards for renderer boundaries, and focused vertical verification before adding broad UI surface area.

- Production retiming/speed curves and advanced effects are Phase 18 scope, though Phase 15 must leave typed extension points.
- Mobile/server audio output and cloud render transport are later binding/runtime phases.
- Large Jianying UI alignment remains an ongoing UI workstream; Phase 15 should improve audio playback/waveform production behavior without turning into a full visual redesign phase.

## Summary

Phase 15 should introduce Rust-owned audio semantics before native playback. The existing code already has segment volume, track mute, volume keyframes, audio dirty domains, render-graph audio mix intent, and FFmpeg export audio filters, but it lacks typed pan/fade/effects carriers and an independent low-latency preview graph. [VERIFIED: crates/draft_model/src/timeline.rs; VERIFIED: crates/draft_commands/src/delta.rs; VERIFIED: crates/render_graph/src/graph.rs; VERIFIED: crates/ffmpeg_compiler/src/filters.rs]

The safest plan is a two-layer Rust split: `audio_engine` [ASSUMED] owns DSP timeline planning, deterministic mix evaluation, preview session state, buffer status, telemetry, and output traits; `audio_output_desktop` [ASSUMED] owns the concrete desktop output implementation using `cpal` [ASSUMED]. This keeps the renderer limited to commands/status and avoids moving native device handles, raw buffers, or FFmpeg audio syntax into TypeScript. [VERIFIED: crates/bindings_node/src/realtime_preview_service.rs; VERIFIED: apps/desktop-electron/src/main/realtimePreviewHost.ts; CITED: https://docs.rs/cpal/latest/cpal/]

**Primary recommendation:** Build a Rust audio graph and DSP evaluator first, gate all playback/buffer responses by `PlaybackGeneration`, then wire a mock output and `cpal` [ASSUMED] CoreAudio/WASAPI backend behind the same trait with CI tests that do not require physical audio devices. [VERIFIED: crates/realtime_preview_runtime/src/session.rs; CITED: https://github.com/RustAudio/cpal]

## Architecture Patterns

### System Architecture Diagram

```text
Accepted draft commands
  -> CommandDelta(Audio domains + dirty ranges)
  -> audio_engine::DspTimelinePlan
      -> typed segment gain / mute / pan / fades / volume keyframes / future effect slots
      -> integer us <-> sample index mapping with rational sample/frame rates
      -> deterministic AudioMixIntent
          -> Preview path: AudioPreviewSession + TimelineClock + PlaybackGeneration
              -> decode/read source audio ranges via media_runtime traits
              -> fill bounded audio buffers
              -> stale/canceled/generation-mismatched buffers rejected
              -> audio_output_desktop::OutputDevice trait
                  -> cpal CoreAudio/WASAPI stream or mock CI stream
          -> Export path: render_graph/ffmpeg_compiler consumes AudioMixIntent
              -> FFmpeg filter plan for final export
              -> parity diagnostics classify preview/export differences

Artifact store waveform/peak status
  -> safe binding summaries
  -> renderer timeline display only
  -> never canonical DSP input
```

**What:** Audio buffer/status results should include target time, generation, canceled/stale flags, diagnostics, and telemetry like preview frames. [VERIFIED: crates/realtime_preview_runtime/src/request.rs]
**When to use:** Every audio render/read/buffer request crossing async boundaries. [VERIFIED: crates/realtime_preview_runtime/src/session.rs]

### Anti-Patterns to Avoid

- **Renderer-owned audio math:** TypeScript must not compute gain curves, pan, fade envelopes, sample positions, graph nodes, or mix buffers; existing guards already reject renderer-owned timeline/keyframe/render semantics and should be extended for audio. [VERIFIED: scripts/phase11-source-guards.sh; VERIFIED: scripts/phase14-source-guards.sh]
- **FFmpeg preview dependency:** Interactive supported audio preview must not call FFmpeg per frame/segment; FFmpeg remains export/fallback/probe where existing crates require it. [VERIFIED: .planning/phases/15-audio-engine-and-dsp-timeline-pipeline/15-CONTEXT.md]
- **Waveform-as-truth:** Waveform and peak data are display artifacts and must not drive DSP gain/pan/fade decisions. [VERIFIED: crates/artifact_store/src/generation.rs; VERIFIED: AGENTS.md]
- **Unbounded buffering:** Audio callbacks must use bounded buffers/status and classify underrun/overflow; unbounded Vec growth in response to renderer commands is a DoS risk. [ASSUMED]

## Common Pitfalls

### Pitfall 1: Duplicating Timeline Clocks

**What goes wrong:** Audio keeps playing buffers from a previous seek/edit while video preview rejects stale frames. [VERIFIED: crates/realtime_preview_runtime/src/session.rs]
**Why it happens:** Audio uses its own generation counter or wall-clock state. [ASSUMED]
**How to avoid:** Store and compare `PlaybackGeneration` on every audio buffer request/result, and advance generation through the same session events as preview. [VERIFIED: crates/realtime_preview_runtime/src/clock.rs]
**Warning signs:** Audio status lacks generation, stale rejection counts, or cancel diagnostics. [VERIFIED: crates/realtime_preview_runtime/src/telemetry.rs]

### Pitfall 2: Treating Waveforms As Audio Semantics

**What goes wrong:** Regenerating a waveform changes preview/export audio behavior or missing waveform artifacts break playback. [ASSUMED]
**Why it happens:** The DSP evaluator reads peak JSON instead of decoded/source audio plus draft semantics. [ASSUMED]
**How to avoid:** Use artifact waveform/peak data only for UI display and keep canonical gain/pan/fade/effect data in draft semantics. [VERIFIED: crates/artifact_store/src/generation.rs; VERIFIED: AGENTS.md]
**Warning signs:** `project.json` schema or generated `Draft.ts` contains waveform paths/peaks or renderer reads artifact DB/blob paths. [VERIFIED: scripts/phase14-source-guards.sh]

### Pitfall 3: Device-Dependent CI

**What goes wrong:** CI fails because no output device, OS mixer, permission, or native backend is available. [CITED: https://docs.rs/cpal/latest/cpal/]
**Why it happens:** Tests require actual CoreAudio/WASAPI stream playback. [ASSUMED]
**How to avoid:** Put capability enumeration and stream construction behind traits; default CI to mock output and env-gate native audio proof like `VIDEO_EDITOR_TEST_NATIVE_MEDIA` does for native media IO. [VERIFIED: crates/media_runtime_desktop/tests/macos_media_io.rs; VERIFIED: crates/media_runtime_desktop/tests/windows_media_io.rs]
**Warning signs:** Non-macOS tests attempt CoreAudio, non-Windows tests attempt WASAPI, or tests require speakers/headphones. [VERIFIED: crates/media_runtime_desktop/tests/macos_media_io.rs; VERIFIED: crates/media_runtime_desktop/tests/windows_media_io.rs]

### Pitfall 4: Preview/Export Drift

**What goes wrong:** Preview applies pan/fades/keyframes one way while export uses FFmpeg filters another way. [VERIFIED: crates/ffmpeg_compiler/src/filters.rs]
**Why it happens:** Preview sample math and export filter compilation are implemented from separate interpretations of draft fields. [ASSUMED]
**How to avoid:** Create a shared `AudioMixIntent` and compare intent/sample summaries before media-binary parity tests. [VERIFIED: .planning/phases/15-audio-engine-and-dsp-timeline-pipeline/15-CONTEXT.md]
**Warning signs:** `ffmpeg_compiler` reads raw `Segment` fields directly for new DSP semantics instead of consuming audio mix intent. [VERIFIED: crates/ffmpeg_compiler/src/filters.rs]
