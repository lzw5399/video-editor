# Phase 15.2 Research: Real GPU Realtime Compositor Closure

## Findings

### Phase 15.1 Playback Was A Baseline Bridge

Phase 15.1 added visible playback evidence and removed the repeated
`requestPreviewFrame` playback loop. The production path on macOS is currently a
native `AVPlayerLayer` child view. This proves a single video can visibly move,
but it bypasses render graph composition and cannot represent text overlays,
image overlays, multi-track stacking, transform, crop, filters, transitions, or
other edited timeline semantics.

### The Runtime Has Contracts But Not The Product Surface

The realtime runtime already has session, clock, generation, render graph
preparation, capability diagnostics, frame providers, texture cache contracts,
and `wgpu` device bootstrap. The compositor module consumes render graph data,
but its current draw path writes CPU RGBA pixels. That is useful as a contract
test but cannot be accepted as final product GPU presentation.

### The Binding Labels Need Tightening First

`NativePreviewPresentationBackend::Gpu` currently describes native composited
playback rather than render-graph GPU composition. Product E2E reads this as
`backend: "gpu"`, which hides the difference. The first plan must split
evidence taxonomy so tests can fail honestly before the final compositor is
connected.

### Product Completion Requires User-Level Cases

The user explicitly wants a reusable integration-test workflow that behaves like
a normal editor user. Rust tests remain necessary for contracts, but every
visible product behavior needs Playwright/Electron evidence: import, drag/add,
play, seek, edit, save/reopen, export, and visible preview/output assertions.

### Visible UI Is Part Of The Contract

If the default UI exposes a feature, users reasonably expect it to work. During
this phase, unsupported controls should be hidden or gated with productized
unavailable states. Leaving a visible control that silently falls back or does
nothing is a product failure even if the underlying code is clean.

## Implementation Direction

1. Split evidence names:
   - `nativeVideo` or equivalent for `AVPlayerLayer` single-video presentation.
   - `gpuCompositor`/`renderGraphGpu` for the final render-graph compositor.
   Product gates must require the latter.
2. Make no-fallback guards reject native-video evidence as Phase 15.2 success.
3. Implement or expose real `wgpu` compositor output for supported render graph
   visual layers and text. CPU readback may be used for test diagnostics, not as
   product success evidence.
4. Feed imported material frames into the compositor through Rust-owned media IO
   and frame/texture handles without per-frame FFmpeg artifact generation.
5. Present compositor output in the desktop preview host and drive play/pause,
   seek, scrub, and generation tracking through realtime runtime.
6. Expand Playwright tests into a product matrix that locks mainstream editing
   behavior end to end.

## Post-Plan-04 RED Finding

The first desktop product E2E for Plan 04 correctly failed after requiring
`renderGraphGpu` and `renderGraphGpuComposited` evidence. The failure is not a
small UI bug: the runtime still needs two explicit prerequisites before the
desktop host can truthfully present imported video through the render-graph GPU
path.

1. Native WGPU surface presentation: the compositor must render to the visible
   desktop preview surface and present without using offscreen CPU readback as
   product evidence.
2. Native decoded texture import: media IO texture handles must resolve to live,
   generation-checked decoded GPU texture leases that the WGPU compositor can
   sample. Metadata-only opaque handle IDs are not enough.

Plan 15.2-04 should not be retried as a desktop host integration task until
Plans 15.2-03A and 15.2-03B are complete.

## Post-Plan-04 Blocker

Plan 15.2-04 completed useful prerequisite work but did not complete product
playback. It added an NV12 WGPU external texture plane sampling contract and
removed the normal desktop route through the legacy native video presenter. The
remaining gap is a Rust-owned realtime playback scheduler:

1. store accepted draft snapshots, preview profile, compositor surface state,
   media registrations, native texture registry, timeline clock, and playback
   generation;
2. on play/seek/scrub ticks, resolve target time, decode/request the material
   frame, validate sampleable texture leases, build render graph intent from
   Rust semantics, and call surface presentation;
3. record `renderGraphGpuComposited` evidence only after presentation succeeds;
4. keep Electron renderer/main as attach/drive/display surfaces, not owners of
   render graph, GPU commands, backend selection, native handles, or fallback.

Plan 15.2-04A owns this scheduler closure. Plans 15.2-05 and 15.2-06 must not
run before 04A proves normal product playback through the compositor path.

## Post-Plan-04A Blocker

Plan 15.2-04A proved the correct fail-closed behavior but not product playback
success. The normal user journey now reaches the Rust-owned scheduler, uses the
native texture bridge, rejects fallback routes, and then fails when WGPU surface
acquisition reports `surface is occluded`.

The next work is not an E2E assertion change and not a fallback policy change.
Plan 15.2-04B must repair the macOS/Electron native surface attachment so the
WGPU surface is visible/non-occluded and can produce a real `SurfaceTexture`.
If the current BrowserWindow native handle is not the right parent for a child
WGPU surface, the main-process native parent acquisition path must change.
Only a successful `present_to_surface` may emit `renderGraphGpuComposited`.

## Post-Plan-04B Blocker

Plan 15.2-04B preserved the correct fail-closed behavior but did not unblock
product playback. It added product RED coverage for occluded surface acquisition
and attempted a retained AppKit child-window/CAMetalLayer attachment, yet the
normal Electron user journey still reaches `surface.get_current_texture()` and
receives `surface is occluded`.

The remaining work is a native drawable lifecycle problem, not an evidence
taxonomy problem. Plan 15.2-04C must make the selected macOS/Electron surface
actually drawable to WGPU before acquire, including validating the parent handle
choice, AppKit/CoreAnimation attachment, visibility/bounds, transaction/runloop
commit, and presentation timing. If the child-window/CAMetalLayer attachment
cannot be made drawable, 04C must replace that attachment strategy. No fallback
route may be added or accepted.

## Post-Plan-04C Blocker

Plan 15.2-04C narrowed the failure further but still did not unblock product
playback. Diagnostics now show the final attachment strategy was a top-level
overlay window with nonzero bounds and an attached `CAMetalLayer`, yet AppKit
reported both the Electron parent window and the overlay window as
`occlusionVisible=false`. That means the next repair must address foreground
activation/active-space state before WGPU acquire, and should stop treating a
free-floating overlay window as the target product architecture.

Plan 15.2-04D should make macOS foreground activation explicit, then use an
Electron-window native subview/layer surface aligned to the preview host rect.
This matches the product goal better than an independent overlay: the WGPU
surface belongs to the editor window and the renderer remains only a
transparent control/display layer. Success still requires real WGPU presentation
and visible compositor playback.

## Post-Plan-04E Blocker

Plan 15.2-04E proved that CDP attachment and main-process observation are not
the remaining blocker. The packaged app launches, the renderer is reachable, and
the product user journey reaches native WGPU surface acquire. The failure still
occurs before present: AppKit reports the parent and child WGPU windows as
occlusion-invisible.

A follow-up probe narrowed the foreground problem further. System Events can
identify the `Video Editor` process by pid and reports it as visible, but it
still reports `frontmost=false` and `windows=0` after activation attempts while
another app remains frontmost. CoreGraphics window-server inspection can report
onscreen windows for the same pid. Plan 15.2-04F must therefore distinguish
three separate facts before WGPU acquire: process activation/frontmost state,
AX/System Events window visibility, and CoreGraphics window-server visibility.

The 04F investigation also flagged a likely geometry bug: renderer host bounds
are CSS pixels/AppKit points, while the current macOS attachment divides
`x/y/width/height` by scale before setting the AppKit view/window frame. Retina
2x diagnostics show a 320x179 drawable paired with an approximately 160x89.5
AppKit view. 04F should keep point-space frames in AppKit and use scale only
for `CAMetalLayer.contentsScale` plus backing-pixel `drawableSize`.

## Risks

- A true native `wgpu` surface bridge is platform-specific. macOS is the current
  host for this work; unsupported platforms must fail closed until implemented.
- Existing tests may pass because they assert `backend: "gpu"` too broadly.
  Tightening evidence is expected to create failing tests before the compositor
  path is connected.
- Text parity must use the existing bundled font registry. Reintroducing system
  font dependency would break preview/export parity.
- CPU/offscreen readback is useful for deterministic assertions but must not be
  the product preview surface.
