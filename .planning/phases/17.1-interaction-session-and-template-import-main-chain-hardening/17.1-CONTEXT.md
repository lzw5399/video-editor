# Phase 17.1: Interaction Session And Template Import Main-Chain Hardening - Context

**Gathered:** 2026-06-24
**Status:** Ready for planning
**Source:** User direction, current-main code inspection, Phase 17 template import results, and production architecture review.

<domain>
## Phase Boundary

Phase 17.1 hardens the editor's main interactive editing chain before mobile/server work. The phase separates generic core interaction capability from the Kaipai adapter, adds a Rust-owned interaction-session path for high-frequency editing, and closes template-import response/invalidation gaps that currently make the UI guess which preview state to refresh.

This phase is not a Kaipai fidelity phase and not a mobile/server binding phase. Kaipai remains an external adapter that produces canonical draft semantics plus an adaptation report. The main-chain work here must improve all supported editing surfaces: preview canvas, timeline, inspector, playhead, keyframes, and template report navigation.

</domain>

<decisions>
## Implementation Decisions

### Scope And Ownership
- **D-01:** Core interaction capability and Kaipai adapter behavior are separate. Interaction sessions are provider-neutral editor infrastructure; Kaipai only uses the same project-session/view-model/report contracts after import.
- **D-02:** The phase targets production-grade rendering and interaction quality, not one-off compatibility with old Kaipai integration code.
- **D-03:** Rust owns draft/timeline/selection/validation/provisional state/preview invalidation/final commit semantics. React may render local pointer affordances, but it must not become the semantic source of truth.
- **D-04:** Commit is the only point that mutates canonical `.veproj/project.json`, increments the project revision, persists the draft, and pushes one undo item.
- **D-05:** Live interaction updates may produce Rust-validated provisional view models and preview deltas, but they must not save, increment revision, or push undo/redo state.
- **D-06:** Cancel and stale rejection leave draft, revision, undo/redo state, and project file unchanged.
- **D-07:** Commit succeeds only when current project revision matches the interaction `baseRevision`; otherwise Rust rejects the session as stale.

### High-Frequency Interaction Definition
- **D-08:** A high-frequency interaction is any user operation that can emit many samples before a semantic commit: pointer drag, slider drag, color picker drag, wheel/trackpad scrub, keyboard repeat stepping, text composition, or rapid report navigation.
- **D-09:** High-frequency does not mean every operation becomes a draft mutation. Some interactions are navigation-only or UI-only, but they still need coalescing/backpressure if they drive preview seek/selection.
- **D-10:** Discrete commands remain normal project-session commands: import, add, split, delete, undo, redo, save, export, track add/rename/lock/mute toggles, fit-mode select, visibility toggle, and one-click keyframe add/remove.

### Production Interaction Session
- **D-11:** Add a Rust-owned `ProjectInteractionSession` state machine: `Idle -> Begun -> Streaming -> CommitPending -> Committed`, with `Begun/Streaming -> Canceled` and `Begun/Streaming/CommitPending -> StaleRejected`.
- **D-12:** The IPC contract should be explicit: `beginProjectInteraction`, `updateProjectInteraction`, `commitProjectInteraction`, and `cancelProjectInteraction`.
- **D-13:** Every update carries monotonic sequence numbers. Rust accepts ordered samples, coalesces obsolete samples, rejects stale interaction IDs/generations, and returns the accepted/coalesced sequence.
- **D-14:** Renderer should throttle outbound updates to requestAnimationFrame and keep at most one update in flight per interaction. Rust remains authoritative; the renderer cannot decide legality.
- **D-15:** The preview runtime should consume provisional interaction deltas/snapshots with generation IDs and reject stale frames.
- **D-16:** Interaction updates must use the scheduler/backpressure model already used by `InteractivePreview` and `ScrubSeek`, where obsolete samples can be coalesced.

### Specific High-Frequency Interactions In Scope
- **D-17:** Preview canvas interactions: move selected segment/text overlay, rotate handle, scale/resize handles, crop handles when exposed, text box/layout resize, and drag-to-position for imported template/text overlays.
- **D-18:** Timeline interactions: segment drag move, cross-track move, left/right edge trim, multi-select move/trim where exposed, transition/fade handles where exposed, and snap/magnet preview during drag.
- **D-19:** Playhead interactions: ruler click-drag, playhead handle drag, timecode scrub, previous/next frame repeat, wheel/trackpad scrub, and rapid report navigation seeks.
- **D-20:** Inspector interactions: visual position/scale/rotation/opacity/crop sliders and numeric steppers, text size/line height/letter spacing/text box/layout/color controls, audio gain/pan/fade sliders, and canvas background/color/aspect fields where previewable.
- **D-21:** Keyframe interactions: dragging keyframe markers in time, dragging value handles or property controls while keyframe mode is active, multi-keyframe move where exposed, and nearest-keyframe deletion/navigation rather than exact-playhead-only deletion.
- **D-22:** Template report interactions: selecting adaptation report rows should focus the related segment/material/track/resource, seek to the relevant time when available, and request preview through coalesced navigation rather than ad hoc UI guesses.

### Template Import Main-Chain Hardening
- **D-23:** `ProjectSessionTemplateImportResponse` must follow the same response shape as other timeline mutations: view model, revision, events, `CommandDelta`, bundle path, project JSON path, and adaptation report.
- **D-24:** UI preview invalidation must be driven by Rust `CommandDelta` and invalidation consumers, not by a hardcoded TypeScript switch over intent names.
- **D-25:** Template import report UI must not expose raw formula JSON, provider URLs, tokens, or backend-only diagnostics. It should be a product report with actionable focus/navigation.

### UI Corrections
- **D-26:** Inspector units must be normalized for users: fade durations in seconds, scale/opacity/crop in percent, rotation in degrees, and no raw microseconds/millis in default product UI.
- **D-27:** Preview chrome should be draft-aware, hide or enable inert controls, preserve product status line visibility, and make selection/rotate/scale handles discoverable.
- **D-28:** Placeholder feature surfaces for sticker/effect/transition/filter/adjustment must either ship real MVP interactions or be clearly gated/hidden so they do not appear functional.
- **D-29:** Timeline trim and keyframe hit areas must be large enough for product use; trim handles should have at least a 10px hit target.

### the agent's Discretion
- Exact crate/module names are flexible. Likely locations include `bindings_node::project_session_service` for session orchestration, `draft_commands` for provisional command evaluation, `draft_model` for interaction/delta contracts, and desktop `nativeBinding.ts`/preload/renderer files for IPC.
- The renderer can keep lightweight local pointer affordances during the same animation frame, but Rust must provide the authoritative provisional result and commit result.
- The planner may split navigation-only interactions from draft-mutating interactions if the API shape keeps clear semantics and testability.

</decisions>

<specifics>
## Specific Ideas

- Begin with the contract gap that is already visible: template import Rust response lacks `events`/`delta`, while TypeScript models it as a timeline response. Fix this before broader sessions.
- Replace `previewAffectingIntentNeedsRefresh` with a delta consumer so new Rust commands and interactions do not need TypeScript invalidation switches.
- Implement interaction sessions for Rust-validated provisional state, not commit-on-release-only ghost state.
- Keep a fast local visual affordance only as a rendering optimization; it must be reconciled with Rust provisional updates and canceled on stale rejection.
- Use one undo item per committed interaction, even if hundreds of update samples were streamed.
- Test for the negative paths: cancel no-op, stale revision rejected, update sequence ordering, no save/revision during live updates, and no high-frequency path through `executeProjectIntent`.

</specifics>

<deferred>
## Deferred Ideas

- Pixel-perfect Kaipai/Jianying parity and proprietary effect reproduction remain deferred to production effects phases.
- Full mobile/server binding architecture remains Phase 18 after 17.1.
- Advanced curve editors, full retiming, complex transitions, masks/blends, and proprietary native effects remain downstream unless their visible controls are already in default UI and must be gated.
- Full text editor composition semantics can be incremental if plain text typing is not yet a default high-frequency preview surface, but per-keystroke undo pollution must not be introduced.

</deferred>
