## Current Chain

### Project Session Commands

`apps/desktop-electron/src/renderer/App.tsx` has a single `executeProjectSessionIntent` path guarded by `commandInFlightRef`. This path is appropriate for discrete commands, but it serializes high-frequency interactions behind a full mutation pipeline. `executeProjectTimelineIntent` then calls `previewAffectingIntentNeedsRefresh`, a TypeScript hardcoded switch over intent names, to decide whether to refresh preview.

### Preview Canvas Drag

`PreviewMonitor.tsx` has local ghost state through `previewDragRef` and `previewDragPreview`. Move and rotate are previewed locally, then `onUpdateSelectedSegmentVisual` fires once on pointer-up. This avoids flooding Rust, but Rust does not validate live samples, preview runtime does not receive provisional state, and invalid drag/crop/rotation behavior is only discovered at commit.

### Timeline Drag/Trim

`Timeline.tsx` has `dragPreview` for move/trim and commits only on pointer-up through `onMoveSelectedSegment` or `onTrimSelectedSegment`. Cross-track drop target calculation is done in the UI, while Rust remains responsible at commit. Live snapping/magnet legality and preview invalidation are not Rust-owned during drag.

### Inspector Sliders

`Inspector.tsx` and `FeaturePanel.tsx` use local form state plus 160ms debounce for visual/audio edits. This can still produce repeated canonical commands, saves, revisions, and preview refreshes while dragging a slider. It also couples UX responsiveness to command roundtrips.

### Playhead/Preview Seek

`handleSeekPlayhead` updates local playhead, syncs project-session playhead, seeks realtime preview host, and seeks audio preview. It is already closer to a navigation interaction, but it lacks a unified session/generation model with other high-frequency interactions.

### Template Import Response

Rust `ProjectSessionTemplateImportResponse` currently serializes `session_id`, `revision`, `viewModel`, `adaptation_report`, `bundle_path`, and `project_json_path`, but no `events` or `delta`. TypeScript models template import as `ProjectSessionTimelineIntentResponse & { adaptationReport }`, which means the binding contract is inconsistent and UI invalidation cannot be driven uniformly.

## Production Target

The production chain should be:

```text
UI pointer/slider/scrub samples
  -> beginProjectInteraction(expectedRevision, kind, selection, playhead, source)
  -> updateProjectInteraction(sequence, sample)
     -> Rust validates against base revision and current session generation
     -> Rust computes provisional draft/view model/overlay/delta
     -> scheduler coalesces obsolete preview/scrub work
     -> preview runtime presents provisional generation and rejects stale frames
  -> commitProjectInteraction(finalSequence)
     -> one canonical draft mutation
     -> one revision increment
     -> one save/autosave decision
     -> one undo item
     -> one CommandDelta-driven invalidation response
```

For navigation-only interactions:

```text
UI navigation samples
  -> coalesced project navigation / preview seek
  -> Rust-owned selection/playhead target and preview generation
  -> no draft save, no undo, no canonical revision unless selection semantics require session revision
```

## Recommended API Shape

```ts
beginProjectInteraction({
  sessionId,
  expectedRevision,
  kind,
  selectionHandles,
  playheadUs,
  pointerSpace,
  source
}) -> { interactionId, baseRevision, interactionGeneration, acceptedSequence: 0 }

updateProjectInteraction({
  interactionId,
  sequence,
  sample
}) -> {
  acceptedSequence,
  coalescedThrough,
  provisionalViewModel,
  provisionalDelta,
  previewTargetUs,
  diagnostics
}

commitProjectInteraction({ interactionId, finalSequence }) -> ProjectSessionIntentResponse

cancelProjectInteraction({ interactionId, reason }) -> { canceled: true, revisionUnchanged: true }
```

## Validation Architecture

### Rust Unit And Binding Tests

- State machine transition tests for begin/update/commit/cancel/stale rejection.
- No revision/save/undo during updates; exactly one revision/save/undo at commit.
- Cancel leaves project file and undo/redo unchanged.
- Commit rejects when current revision differs from `baseRevision`.
- Sequence ordering rejects duplicate/out-of-order samples and reports `acceptedSequence`.
- Coalescing stores latest pending sample and drops obsolete intermediate updates.
- Multi-select timeline move/trim is atomic.
- Cross-track legality validates track type, lock, and bounds.
- Crop/scale/rotation bounds stay in canonical integer units.
- Keyframe edit preserves segment-relative microsecond math.
- Binding tests cover unknown fields, stale interaction ID, stale revision, and template import response `events`/`delta`.

### Frontend And Product E2E

- Playwright tests for preview move/rotate/scale, timeline move/trim/cross-track, inspector visual/audio/text slider drag, playhead scrub, keyframe marker edit, and template report row navigation.
- Screenshots at `1280x800` and `1120x720` for inspector units, preview chrome, report panel hierarchy, and timeline hit areas.
- Source guards fail any high-frequency draft mutation routed through old repeated `executeProjectIntent` loops.
- Source guards fail hardcoded TypeScript invalidation switches for new Rust command deltas.
- Product preview evidence must show compositor state changes with edit state and reject DOM-only ghost success.

## Risks

- If the UI keeps local ghost preview as the only live feedback, interaction will feel fast but semantically wrong under snapping, crop bounds, keyframes, cross-track moves, and preview/export parity.
- If every slider sample remains a canonical command, undo/revision/save will become noisy and preview scheduling will be unstable.
- If provisional preview snapshots do not carry generation IDs, stale frames can display after cancel or commit.
- If template import responses remain outside the delta contract, every future import-like command will need a bespoke UI invalidation path.

## Recommendation

Implement Phase 17.1 as six waves:

1. Response/delta contract hardening.
2. Rust interaction-session state machine and binding API.
3. Preview canvas and inspector live interactions.
4. Timeline, playhead, and keyframe live interactions.
5. Template report navigation plus product UI corrections.
6. Aggregate E2E, performance budgets, source guards, and phase verification.

This sequence fixes the main-chain contract first, then moves high-frequency behavior behind Rust-owned provisional semantics without making Kaipai an internal render concept.
