# Phase 17.1 UI Review: High-Frequency Interaction And Template Report Surfaces

**Reviewed:** 2026-06-24  
**Overall score:** 15/24  
**Posture:** Production UI corrections should be done with the interaction architecture, not as isolated CSS polish.

## Follow-Up Audit 2026-06-24

**Read-only subagent audit score:** 12/24.

The second audit found that the current UI is below the Phase 17.1 UI-SPEC bar. These findings are now blocking inputs for Waves 3-6:

- High-frequency editing is still local/debounced or commit-only instead of Rust provisional sessions. `PreviewMonitor.tsx` uses local drag/rotate deltas and commits on pointer-up; `Timeline.tsx` uses local move/trim preview and commits on pointer-up; inspector controls debounce canonical updates.
- Template report rows are static diagnostics. `FeaturePanel.tsx` report rows need click/keyboard navigation, selected row state, canonical target focus/seek, and report-only state for rows without editable draft targets.
- Unsupported controls still look available. Preview view controls and feature showcase panels need to be hidden or explicitly gated with `暂不可用`, disabled state, and no active resource-card styling.
- Product units still leak internal ranges. Scale, opacity, crop, text layout, line height, letter spacing, and fade controls must display percent/seconds/product units.
- Default-mode internal copy risk remains around GPU/scheduler/runtime/resource generation wording. Default users should see recovery-oriented product copy, not backend terminology.
- Hit targets miss the 17.1 contract: timeline trim/playhead/keyframe markers must have effective 16px targets, and keyframe markers must become editable where exposed.
- Typography is not yet aligned to the UI-SPEC 11/12/13/14px and 400/600 contract.
- Existing UI tests do not prove the full design contract. Product interaction tests must fail DOM-only ghost previews, static report rows, and unsupported controls that appear functional.

## Findings

### 1. Inspector Units And Commit Behavior

The inspector exposes several semantically correct internal values but the product UI needs clearer user units. Scale, opacity, crop, text box/layout, fade durations, and color controls should show product units and stable labels. More importantly, visual/audio/text sliders currently rely on local form state and debounce, which can still produce many canonical commands during a drag.

**Required direction:** inspector controls become interaction-session sources. Dragging a slider streams provisional Rust updates; releasing commits one command. Numeric blur/Enter can use direct commit.

### 2. Preview Chrome And Handles

The preview shell has a recognizable monitor structure, but some controls are inert or read like placeholders. Selection/rotate handles exist, but scale/crop affordances are not fully discoverable. Product status should stay visible without debug fallback copy in default mode.

**Required direction:** enable or hide fit/ratio/fullscreen controls; make transform handles discoverable; ensure canvas title and state are draft-aware; verify with screenshots.

### 3. Timeline Drag Surface

Timeline drag and trim have local ghost state, but Rust does not own live snapping/magnet/cross-track legality until commit. Trim handles need product hit areas, and keyframe marker editing is not yet a first-class live interaction.

**Required direction:** wire timeline drag/trim/keyframe marker movement through interaction sessions. Keep local ghost only as a same-frame affordance reconciled by Rust provisional results.

### 4. Template Report Hierarchy

The template adaptation report is useful but mostly static. Users need to click report rows and understand what changed in the draft. Unsupported/missing-resource rows need a report-only state when there is no canonical entity to focus.

**Required direction:** report rows should navigate/focus/seek through a coalesced navigation path and never expose raw provider JSON or provider URLs.

### 5. Placeholder Feature Surfaces

Sticker/effect/transition/filter/adjustment surfaces risk looking functional before production preview/export backing exists.

**Required direction:** ship actual MVP interactions where supported, or hide/gate with product unavailable copy/tooltips. Do not present default visible controls that cannot affect production preview/export.

## UI Acceptance

- Default product UI does not show raw microseconds/millis for user-editable controls.
- Fade durations display seconds.
- Scale, opacity, and crop display percentages.
- Rotation displays degrees.
- Preview transform handles are visible, stable, and have accessible labels/tooltips.
- Timeline trim handles have at least 10px hit target.
- Keyframe deletion/navigation does not require exact playhead equality when the UI presents a nearby focused keyframe.
- Template report row navigation focuses entities or clearly reports that the row has no editable draft target.
- Screenshots at `1280x800` and `1120x720` show no overlapping text, no debug-only copy in default mode, and stable five-zone workspace layout.
