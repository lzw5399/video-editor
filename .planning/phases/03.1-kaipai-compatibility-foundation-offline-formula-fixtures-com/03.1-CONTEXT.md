# Phase 03.1: Kaipai Compatibility Foundation - Context

**Gathered:** 2026-06-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 03.1 establishes the compatibility foundation for Kaipai formula import. It delivers the offline formula-bundle contract, sanitized fixture corpus shape, compatibility report semantics, resource bundle/localizer boundary, source guards, and explicit Draft v2/template-semantics gaps. It does not call live Kaipai APIs, does not implement a full mapper, does not claim pixel-level Kaipai parity, does not build preview/export, and does not make the Android worker a product runtime dependency.

</domain>

<decisions>
## Implementation Decisions

### Offline Formula Bundle Boundary
- **D-01:** The first Kaipai path starts from an already-produced offline formula bundle. Live Kaipai API integration, auth, rate limits, submit/poll behavior, and provider retries are out of this phase.
- **D-02:** A formula bundle must capture the external evidence needed to reproduce and audit import behavior: raw formula, template id, recipe id, formula task/request ids when available, source media metadata, direct material list, recognizer `word_list`, `safe_area` evidence, and resource references.
- **D-03:** The adapter may preserve raw Kaipai JSON as external input/provenance fixture data, but raw formula JSON must not become canonical `.veproj/project.json` render semantics.

### Layer Ownership
- **D-04:** `draft_model`, `engine_core`, `render_graph`, and `ffmpeg_compiler` must not import Kaipai provider code, Android worker code, or raw Kaipai formula interpretation. They consume canonical Jianying-style draft semantics only.
- **D-05:** Formula parsing, provenance, supported/degraded/unsupported mapping, and compatibility report generation belong to an adapter boundary, later likely `adapter_kaipai`.
- **D-06:** Resource download/copy, sha256 validation, local relative paths, and missing-resource diagnostics belong to a resource-localizer service boundary. Renderable resources must not depend on remote template URLs at preview/export time.
- **D-07:** Android worker output may be used as oracle/calibration fixture evidence. It must not be called from Video Editor runtime code.

### `safe_area` Handling
- **D-08:** `safe_area` is provider/preprocess evidence for formula generation and adapter provenance. It is not Rust draft core semantics and must not be calculated by `draft_model`, `engine_core`, `render_graph`, or `ffmpeg_compiler`.
- **D-09:** If a later adapter maps `safe_area` into visible behavior, it must map to concrete canonical draft properties such as canvas adjustment, transform, crop, or placement. The raw evidence remains provenance, not render logic.
- **D-10:** A future `SafeAreaProvider` or ASR compatibility path should be spiked separately. It must not block offline formula import.

### Compatibility Report
- **D-11:** Compatibility reports are a first-class output of external import. They must classify at least `supported`, `degraded`, `unsupported`, `missingResource`, and `needsNativeEffect`.
- **D-12:** Unsupported native effects must be explicit. Do not hide unsupported Kaipai behavior in generic `Filter.parameters` strings just to make an import appear complete.
- **D-13:** Compatibility report snapshots are required before any mapper POC is considered complete.

### Draft And Resource Preparation
- **D-14:** This phase should identify and, if scoped by planning, introduce the smallest general-purpose schema foundations needed by template import: canvas, font material/resource references, resource manifest, canvas adjustment/transform, sticker/text sticker payloads, external provenance, and compatibility report types.
- **D-15:** Schema additions must remain Jianying-style and general-purpose. They should also support later Jianying/CapCut draft import, manual stickers, mobile/server rendering, and preview/export parity.
- **D-16:** If the planner decides schema changes are too large for one phase, it must split them into explicit plans and leave a clear Draft v2/template-semantics follow-up before any offline Kaipai mapper POC.

### Tests And Regression Gates
- **D-17:** Phase plans must define executable gates before implementation starts. Minimum gates: `just test`, schema/TypeScript drift checks, formula fixture validation, compatibility report snapshots, source guards for core/render crates, and resource-localizer tests for local relative paths.
- **D-18:** Fixture data must be sanitized. No session tokens, signed URLs, cookies, account IDs, or production credentials may be committed.
- **D-19:** The fixture corpus should prove shape and behavior with small, deterministic samples before depending on real provider calls.

### the agent's Discretion
- The planner may choose exact crate/module names, but it should keep adapter/provider/resource-localizer concepts separate from canonical draft semantics.
- The planner may decide whether compatibility report types live initially in `draft_model`, a new compatibility crate, or an adapter crate, as long as generated schemas/tests remain clear and raw provider data does not leak into core/render layers.
- The planner may decide whether this phase implements all Draft v2/template foundations or records them as a follow-up, but it must not plan a full mapper POC before those gaps are handled.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `draft_model` already uses serde, schemars, and ts-rs as the canonical schema/TypeScript source. Any new draft/report contracts should follow this generated-contract pattern.
- `project_store` already owns `.veproj/project.json` persistence and material URI classification. Resource-localizer plans should preserve this separation and avoid UI-owned path logic.
- `validation.rs` already rejects derived artifact leakage. Similar guard patterns can protect raw provider data, render graphs, remote template URLs, and Android artifacts from `project.json`.
- `schema_exports.rs` already enforces generated schema/TypeScript drift and should be extended for any new types.

### Established Patterns
- Core semantic crates are platform-free and process-free.
- Service-boundary crates coordinate filesystem/media runtime work after core semantics are defined.
- `.veproj/project.json` stores semantic state only. Derived artifacts and adjacent reports/resources should remain separate.
- Tests should be deterministic and fixture-driven, with positive/negative fixture classification where applicable.

### Integration Points
- Future `adapter_kaipai` or equivalent should consume offline formula bundles and emit canonical drafts plus compatibility reports.
- Resource-localizer work likely connects to `project_store` path helpers but should not move provider logic into `project_store`.
- Draft v2/template semantics must be designed before render graph or FFmpeg compiler consume template-derived visual layers.
- Source guards should scan core/render crates for forbidden Kaipai/Android/provider imports and raw formula dependencies.

</code_context>

<specifics>
## Specific Ideas

- The first visible product win is: choose/get a template formula, convert it to a local editable draft, preview/edit it, and export through our own renderer later. The old Android worker path is too slow and not editable.
- UI remains Chinese on desktop, but this phase is not a UI phase.
- Terminology should stay Jianying-style internally and externally; do not introduce separate internal names for concepts already covered by draft/material/track/segment/sticker/text/filter/transition/keyframe.
- The plan should be strict and incremental: research and fixture shape first, then report/resource foundations, then schema/template semantics, then mapper POC later.

</specifics>

<deferred>
## Deferred Ideas

- Live Kaipai API/provider integration belongs after offline formula bundle import is proven.
- `SafeAreaProvider`, ASR-to-`VoiceResult.word_list`, and non-Android `safe_area` generation need a separate spike.
- Full Kaipai/Jianying/CapCut draft import/export and pixel-level native effect parity are out of this phase.
- Desktop UI surfacing of compatibility reports can be planned later, after report schema and snapshots exist.
- Preview/export parity for template overlays belongs after Phase 5 render graph/FFmpeg foundations or a dedicated follow-up phase.

</deferred>
