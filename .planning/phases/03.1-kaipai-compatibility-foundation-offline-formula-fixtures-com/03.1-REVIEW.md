# Phase 03.1: Code Review Report

**Reviewed:** 2026-06-17T14:51:35Z
**Depth:** standard
**Files Reviewed:** 32
**Status:** issues_found

## Summary

Reviewed the Kaipai adapter contract, compatibility report generation, resource localizer, generated schemas/TypeScript, fixtures, and tests. The current tests pass (`cargo test -p adapter_kaipai`), and broad source-guard scans did not find provider leakage into core/render crates or obvious committed secrets. The remaining defects are contract and edge-case failures: unsupported formula semantics can be reported as supported, common secret keys are not blocked, resource localization is vulnerable to symlink escape, and resource copies can corrupt each other.

## Narrative Findings (AI reviewer)

## Critical Issues

### CR-01: Formula sanitizer misses common credential keys

**Classification:** BLOCKER
**File:** `crates/adapter_kaipai/src/formula_bundle.rs:335`
**Issue:** `is_credential_like_key` only rejects a small set of exact normalized names. It blocks `access_token` and `Cookie`, but allows equally sensitive keys such as `apiKey`, `api_key`, `password`, `secretKey`, `privateKey`, `bearerToken`, and `authorizationHeader`. Because `formula` is preserved as raw `serde_json::Value`, those fields can pass `KaipaiFormulaBundle::from_json_value`, be committed in fixtures, and leak provider credentials.
**Fix:**
```rust
fn is_credential_like_key(key: &str) -> bool {
    let normalized = key
        .chars()
        .filter(|character| character.is_ascii_alphanumeric())
        .flat_map(char::to_lowercase)
        .collect::<String>();

    const TERMS: &[&str] = &[
        "apikey", "password", "token", "authorization", "auth",
        "cookie", "session", "secret", "signature", "credential",
        "privatekey", "bearer", "account",
    ];

    TERMS.iter().any(|term| normalized.contains(term))
}
```
Add negative tests for `apiKey`, `password`, `secretKey`, and `authorizationHeader`.

### CR-02: Resource localization follows symlinks outside trusted roots

**Classification:** BLOCKER
**File:** `crates/adapter_kaipai/src/resource_localizer.rs:122`
**Issue:** The localizer checks that the resource URI string has no `..`, scheme, or absolute prefix, then reads/copies `source_root.join(path)` directly. If an offline formula bundle contains `resources/fonts/redacted.ttf` as a symlink to a file outside the bundle, `exists`, `fs::read`, and `fs::copy` follow it. The same issue exists on the destination side when copying into an existing `.veproj` tree with symlinked parents. This bypasses the path traversal guard and can copy private local files into the project bundle or overwrite files outside the bundle.
**Fix:**
```rust
let canonical_source_root = request.source_root.canonicalize()?;
let source_path = source_path_for_uri(&request.source_root, source_uri).ok_or(...)?;
let canonical_source = source_path.canonicalize()?;
if !canonical_source.starts_with(&canonical_source_root) {
    return Ok(failed_resource(resource, LocalizedResourceStatus::UnsafePath, None));
}
if fs::symlink_metadata(&source_path)?.file_type().is_symlink() {
    return Ok(failed_resource(resource, LocalizedResourceStatus::UnsafePath, None));
}
```
Apply the same canonical-root/no-symlink policy to `bundle_path` and the destination parent before `fs::copy`.

### CR-03: Unknown or nested formula semantics can be reported as supported

**Classification:** BLOCKER
**File:** `crates/adapter_kaipai/src/compatibility_report.rs:136`
**Issue:** `classify_formula_bundle_compatibility` only probes `formula.textStyleFallback`, top-level `formula.unsupportedBlocks`, and top-level `formula.effects`. Any unsupported block, native effect, sticker/text effect, transform, or provider-specific field nested under realistic formula structures is ignored. When no item is collected, lines 141-156 emit a `supported` material item, so a formula with unclassified semantics can be falsely reported as supported.
**Fix:** Make support an allowlisted outcome, not the default. Either parse a typed supported-subset model or recursively walk the raw formula and emit `unsupported`/`degraded` for every unrecognized provider block before adding a supported item.
```rust
let unsupported = collect_unclassified_formula_paths(&bundle.formula);
for path in unsupported {
    items.push(unsupported_formula_item(path));
}
if items.is_empty() && formula_matches_supported_subset(&bundle.formula) {
    items.push(supported_source_material_item(bundle));
}
```
Add fixtures with nested `timeline.segments[].effects[].nativeEffectId` and unknown nested formula blocks.

### CR-04: Missing-resource reporting is a placeholder heuristic, not resource validation

**Classification:** BLOCKER
**File:** `crates/adapter_kaipai/src/compatibility_report.rs:260`
**Issue:** `collect_missing_resource_items` classifies missing resources only when the ID starts with `missing-` or the URI contains `/missing/`. A normal resource path that is absent from the offline bundle produces no report item, while a valid resource path containing `/missing/` is reported missing even if localization succeeds. This makes the compatibility report disagree with the localizer and can claim a bundle is supported despite unavailable renderable resources.
**Fix:** Remove name-based missing-resource detection from `classify_formula_bundle_compatibility`. Run `ResourceLocalizer` first and merge its `diagnostics` into the report, or pass a validated localization result into the report builder.
```rust
pub fn classify_formula_bundle_compatibility(
    bundle: &KaipaiFormulaBundle,
    localization: Option<&ResourceLocalizationResult>,
    generated_at: impl Into<String>,
) -> CompatibilityReport {
    if let Some(localization) = localization {
        items.extend(localization.diagnostics.iter().cloned());
    }
    // formula classification follows
}
```

### CR-05: Resource destination collisions overwrite localized assets

**Classification:** BLOCKER
**File:** `crates/adapter_kaipai/src/resource_localizer.rs:223`
**Issue:** For resource URIs that do not already start with the expected kind directory, `destination_uri_for_resource` discards the source subdirectory and keeps only `file_name`. For example, `templateA/overlay.png` and `templateB/overlay.png` both localize to `resources/images/overlay.png`; line 168 then lets the later copy overwrite the earlier file while both manifest entries remain `available`. That corrupts imports and can make segments render with the wrong asset.
**Fix:** Preserve a sanitized relative path, include `resource_id` or a content hash in the destination, and track duplicate destinations as errors before copying.
```rust
let destination = PathBuf::from("resources")
    .join(subdir)
    .join(safe_resource_stem(&resource.resource_id))
    .join(file_name);
if !seen_destinations.insert(destination.clone()) {
    return Ok(failed_resource(resource, LocalizedResourceStatus::UnsafePath, None));
}
```

## Warnings

### WR-01: Generated JSON Schema accepts values Rust rejects

**Classification:** WARNING
**File:** `schemas/kaipai-formula-bundle.schema.json:154`
**Issue:** The generated schema permits `sourceMedia.width`, `height`, and `durationMs` to be `0`, and it permits empty strings for required evidence fields. Rust validation rejects those values in `KaipaiFormulaBundle::validate` at `crates/adapter_kaipai/src/formula_bundle.rs:72` and `crates/adapter_kaipai/src/formula_bundle.rs:87`. A UI or external fixture validator using only `schemas/kaipai-formula-bundle.schema.json` can accept a bundle that the adapter later rejects, which is contract drift.
**Fix:** Post-process the generated schema in `crates/adapter_kaipai/tests/schema_exports.rs` to add `minLength: 1` for required string evidence and `minimum: 1` for positive numeric fields, then add negative schema assertions for zero dimensions/duration and empty IDs.
