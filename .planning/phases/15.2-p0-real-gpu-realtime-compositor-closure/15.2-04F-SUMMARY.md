# Phase 15.2 Plan 04F: Packaged macOS WGPU Surface Repair Summary

**Plan 04F is complete. The packaged product journey now imports a repo video, adds it to the timeline, clicks play, and proves `renderGraphGpuComposited` frames advance while the actual screen-captured preview center changes.**

## Accomplishments

- Added foreground/window-server diagnostics to the packaged product harness so failures include System Events and CoreGraphics window evidence.
- Replaced the macOS separate child `NSWindow` presenter with an in-window `NSView` carrying the `CAMetalLayer` used by WGPU.
- Corrected macOS geometry semantics: preview host bounds are AppKit points, while `scaleFactorMillis` only controls the layer contents scale and backing-pixel drawable size.
- Fixed the real WGPU external texture shader for wgpu 29 by binding an explicit sampler and using the three-argument external texture sampling call.
- Fixed AVFoundation native decode to seek to the requested source time instead of repeatedly returning the first sample.
- Fixed realtime compositor sampling to request target-relative source positions instead of each layer's source start for every frame.
- Hid stale preview PNG imagery when the realtime native surface is product-ready.
- Tightened the product E2E to reject fallback evidence and assert macOS system-screenshot preview-center pixels change after a normal play click.

## Root Cause

04E and the first 04F pass had several blockers mixed together:

- The macOS child-window presenter stayed AppKit occlusion-invisible even when the packaged app had CoreGraphics onscreen windows.
- The real WGPU external texture pipeline crashed on wgpu 29 because the shader used the old two-argument `textureSampleBaseClampToEdge` form.
- macOS native decode ignored requested source time, so playback could repeatedly present the first decoded sample.
- The realtime compositor requested video frames at `layer.source_timerange.start` for every graph target time.
- Product E2E could pass on telemetry/region changes without proving the user-visible video pixels moved.

04F fixed those issues: the surface is now an in-window subview with correct point/backing-pixel sizing, the NV12 external texture shader matches the current wgpu API, source-time decode and target-relative compositor sampling are covered by regressions, and product E2E asserts real screen-pixel motion.
