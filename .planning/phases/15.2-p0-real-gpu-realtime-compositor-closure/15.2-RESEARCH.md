# Phase 15.2 Research: Real GPU Realtime Compositor Closure

## Findings

### Phase 15.1 Playback Was A Baseline Bridge

Phase 15.1 added visible playback evidence and removed the repeated
`requestPreviewFrame` playback loop. The production path on macOS is currently a
native `AVPlayerLayer` child view. This proves a single video can visibly move,
but it bypasses render graph composition and cannot represent text overlays,
image overlays, multi-track stacking, transform, crop, filters, transitions, or
other edited timeline semantics.

### The Runtime Has Contracts But Not The Product Surface

The realtime runtime already has session, clock, generation, render graph
preparation, capability diagnostics, frame providers, texture cache contracts,
and `wgpu` device bootstrap. The compositor module consumes render graph data,
but its current draw path writes CPU RGBA pixels. That is useful as a contract
test but cannot be accepted as final product GPU presentation.

### The Binding Labels Need Tightening First

`NativePreviewPresentationBackend::Gpu` currently describes native composited
playback rather than render-graph GPU composition. Product E2E reads this as
`backend: "gpu"`, which hides the difference. The first plan must split
evidence taxonomy so tests can fail honestly before the final compositor is
connected.

### Product Completion Requires User-Level Cases

The user explicitly wants a reusable integration-test workflow that behaves like
a normal editor user. Rust tests remain necessary for contracts, but every
visible product behavior needs Playwright/Electron evidence: import, drag/add,
play, seek, edit, save/reopen, export, and visible preview/output assertions.

### Visible UI Is Part Of The Contract

If the default UI exposes a feature, users reasonably expect it to work. During
this phase, unsupported controls should be hidden or gated with productized
unavailable states. Leaving a visible control that silently falls back or does
nothing is a product failure even if the underlying code is clean.

## Implementation Direction

1. Split evidence names:
   - `nativeVideo` or equivalent for `AVPlayerLayer` single-video presentation.
   - `gpuCompositor`/`renderGraphGpu` for the final render-graph compositor.
   Product gates must require the latter.
2. Make no-fallback guards reject native-video evidence as Phase 15.2 success.
3. Implement or expose real `wgpu` compositor output for supported render graph
   visual layers and text. CPU readback may be used for test diagnostics, not as
   product success evidence.
4. Feed imported material frames into the compositor through Rust-owned media IO
   and frame/texture handles without per-frame FFmpeg artifact generation.
5. Present compositor output in the desktop preview host and drive play/pause,
   seek, scrub, and generation tracking through realtime runtime.
6. Expand Playwright tests into a product matrix that locks mainstream editing
   behavior end to end.

## Risks

- A true native `wgpu` surface bridge is platform-specific. macOS is the current
  host for this work; unsupported platforms must fail closed until implemented.
- Existing tests may pass because they assert `backend: "gpu"` too broadly.
  Tightening evidence is expected to create failing tests before the compositor
  path is connected.
- Text parity must use the existing bundled font registry. Reintroducing system
  font dependency would break preview/export parity.
- CPU/offscreen readback is useful for deterministic assertions but must not be
  the product preview surface.
