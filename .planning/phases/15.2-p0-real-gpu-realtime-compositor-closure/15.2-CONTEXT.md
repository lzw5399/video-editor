# Phase 15.2: P0 Real GPU Realtime Compositor Closure - Context

**Gathered:** 2026-06-20
**Status:** Ready for planning
**Source:** User P0 correction after Phase 15.1 verification

<domain>
## Phase Boundary

Phase 15.2 exists because Phase 15.1 proved a baseline single-video native
playback path, but that is not the final editor preview path. A production video
editor preview must render the edited timeline through the same Rust-owned
semantics that drive export: draft commands -> normalized timeline -> resolved
frame state -> render graph -> realtime compositor -> visible preview.

This phase closes the product gap for mainstream editing behavior exposed in
the default UI. It must prove the behavior from normal user actions, not just
unit tests or bindings.
</domain>

<decisions>
## Locked Decisions

- Phase 15.2 is the current P0 before Jianying-style UI convergence and Phase 16.
- Product playback success cannot be satisfied by `AVPlayerLayer` single-video
  playback, first-frame snapshots, preview PNG/FFmpeg artifacts, mock/offscreen
  displays, decoded CPU hashes, or DOM overlays.
- A native single-video player layer may remain only as an explicit unsupported
  bridge or diagnostic while the compositor path is being implemented. It must
  not be labeled as realtime GPU composition.
- Every default visible editing control must either work through the product
  E2E matrix or be hidden/gated until the implementation exists.
- E2E tests must use repository-owned fixtures and perform real user workflows.
- Fallback paths may report unavailable/degraded diagnostics but cannot continue
  playback, mark preview success, or satisfy regression gates.
- Because this is a greenfield editor, refactors must remove or gate obsolete
  legacy/fallback paths instead of preserving them as compatibility behavior.
  Reviewers must check `docs/refactor-and-legacy-cleanup-policy.md`.
- Renderer code remains a command/display surface. Rust owns draft semantics,
  render graph preparation, compositor classification, frame/texture ownership,
  and export semantics.
- UI icon work for later UI convergence should use assets selected from
  `/Users/zhiwen/code/video-editor/icons` and copy chosen assets into app code.
- Phase 15.3 UI convergence must include a project entry state for creating or
  opening a project before importing materials, and export must move to the
  top-right modal entry point.
- Product UI must not expose normal-user backend/runtime/mock selectors. There
  should be one production preview path for supported playback; diagnostics may
  report why that path is unavailable.
</decisions>

<current_state>
## Current Implementation Facts

- `crates/realtime_preview_runtime/src/gpu/compositor.rs` is named as a GPU
  compositor, but the current drawing implementation composites CPU RGBA pixels
  in Rust loops and returns offscreen pixel buffers.
- `crates/bindings_node/src/native_preview_presenter.rs` uses macOS
  `AVPlayerLayer` for the supported product playback path and intentionally
  rejects multi-segment, text-bearing, filtered, transitioned, animated, or
  otherwise non-single-video compositions.
- `NativePreviewPresentationState::available` reports backend `Gpu`, which made
  the product E2E treat native single-video playback as GPU/composited success.
- `apps/desktop-electron/src/main/realtimePreviewHost.ts` accepts the native
  presentation state as production playback evidence.
- Existing no-product-fallback guards reject decoded/FFmpeg/mock evidence but
  need to be strengthened so native single-video playback cannot be confused
  with render-graph GPU composition.
- Phase 15.2 Plan 04 RED E2E proved that desktop playback is still not product
  complete: the playhead can advance while the visible video is not driven by a
  render-graph GPU compositor. The work is now split before retrying Plan 04:
  Plan 03A adds a native WGPU surface presentation target, Plan 03B imports
  decoded native textures into the WGPU compositor, and Plan 04 then connects
  those runtime capabilities to the Electron product playback scheduler.
- Phase 15.2 Plan 04 added the NV12 WGPU external texture sampling contract and
  removed native-video bridge success, but it intentionally remains blocked for
  normal product playback: there is still no Rust-owned scheduler that decodes
  the current timeline frame, builds frame-provider state, presents to the WGPU
  surface, and records `renderGraphGpuComposited` evidence. Plan 04A owns that
  scheduler closure before the product E2E matrix can expand.
</current_state>

<acceptance>
## Phase Acceptance

The phase is complete only when a normal user can import repo-owned materials,
edit a mainstream timeline, press play, and see the visible preview reflect the
render graph compositor output as time and edits change. The same flow must
cover save/reopen/export for the supported subset, and all success gates must
fail when fallback or non-compositor evidence is used.
</acceptance>
