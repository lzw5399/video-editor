# Phase 15: audio-engine-and-dsp-timeline-pipeline - Context

**Gathered:** 2026-06-19 (inline assumptions mode)
**Status:** Ready for research and planning

<domain>
## Phase Boundary

Phase 15 introduces the production audio preview and DSP timeline foundation for the editor. It must build an independent Rust-owned audio graph synchronized to the existing `TimelineClock` and `PlaybackGeneration` contracts, define typed audio DSP semantics for preview/export parity, connect waveform/peak artifacts as display data only, and prepare platform output seams for macOS CoreAudio and Windows WASAPI without moving audio semantics into Electron.

</domain>

<decisions>
## Implementation Decisions

### Audio Runtime Ownership
- **D-01:** Audio preview playback is Rust-owned. Electron/React may issue play, pause, seek, cancel, and device-selection commands, but it must not construct audio graphs, mix buffers, FFmpeg commands, output-device handles, cache keys, or timeline state.
- **D-02:** The audio runtime must use the existing `TimelineClock` and `PlaybackGeneration` concepts from `realtime_preview_runtime`. Seek, pause, resume, accepted edits, draft reloads, and material relinks must advance/reuse generation rules so stale audio buffers cannot overwrite current playback.
- **D-03:** Preview audio scheduling and buffering must be independent from FFmpeg preview frame generation. FFmpeg may remain an export/fallback/runtime source where existing crates require it, but supported interactive audio preview cannot depend on per-frame/per-segment FFmpeg preview requests.

### DSP Timeline Semantics
- **D-04:** Segment gain, track mute, pan, fades, and keyframed volume must be represented as typed Rust DSP timeline semantics using integer microseconds, frame/sample indices, rational rates, and bounded integer gain units. Do not persist naked floating-point seconds in draft semantics.
- **D-05:** Future audio effects need extension points in the DSP graph, but Phase 15 should only implement the semantic carriers and classifier needed to keep later effects from changing ownership boundaries.
- **D-06:** Draft commands remain the source of accepted audio semantic edits. The audio graph consumes accepted draft state and command deltas; it does not mutate draft state or invent timeline edits.

### Preview, Export, And Parity
- **D-07:** Preview and export audio mixdown must share the same Rust DSP evaluation path wherever possible. Any unavoidable preview/export differences must be classified in parity diagnostics rather than hidden in UI behavior.
- **D-08:** Export audio remains integrated with the existing render/export pipeline, but `ffmpeg_compiler` should consume audio mix intent generated by Rust semantics. The renderer must not assemble FFmpeg filter graphs or arguments for audio.
- **D-09:** Audio parity tests should use deterministic fixtures from `testkit` and compare typed mix intent/sample summaries before relying on heavyweight media binaries.

### Platform Output Boundary
- **D-10:** macOS and Windows output support should be represented behind Rust traits and capability reports. macOS targets CoreAudio and Windows targets WASAPI, but Phase 15 may introduce mock/stub device implementations where direct OS playback is not yet feasible in CI.
- **D-11:** Native audio output handles and buffers must not cross into TypeScript. Bindings expose opaque session IDs, command payloads, safe status summaries, device capability summaries, and diagnostics.

### Waveform And Artifact Store Boundary
- **D-12:** Waveform and peak data from `artifact_store` are derived display artifacts. They can drive timeline UI rendering, but they must not become canonical audio semantics in `.veproj/project.json` or the DSP evaluator.
- **D-13:** The desktop waveform UI should consume safe artifact summaries and display-ready peak payloads through generated command/binding contracts. It must not read SQLite, blob paths, fingerprints, dirty ranges, or artifact roots directly.

### Desktop Product Boundary
- **D-14:** Phase 15 should remove or avoid debug-heavy audio UI. Production-facing desktop UI should show only playback state, waveform availability, safe buffering/status labels, and controls needed for Jianying-style editing.
- **D-15:** The UI direction remains Jianying-style with feature reduction: familiar playback/seek/timeline behavior and compact controls, but no unsupported digital-human or unrelated advanced panels.

### the agent's Discretion
The planner may choose exact crate names and module splits for the audio graph, DSP evaluator, output traits, bindings, and tests. It should prefer small Rust crates with clear ownership, source guards for renderer boundaries, and focused vertical verification before adding broad UI surface area.

</decisions>

<specifics>
## Specific Ideas

- Consider a Rust audio crate such as `audio_engine` or `audio_graph` that owns DSP timeline preparation, graph classification, preview session state, and output traits.
- Reuse `PlaybackGeneration`, `PlaybackRate`, `PlaybackState`, and `TimelineClock` from `realtime_preview_runtime` instead of duplicating clock types.
- Represent gain/pan/fade/keyframed volume as typed structs with integer microsecond ranges and deterministic interpolation helpers.
- Add safe bindings for audio preview commands and status summaries before adding large UI changes.
- Use source guards to reject renderer-owned audio graph construction, sample mixing, output-device handles, FFmpeg filter assembly, artifact path reads, and waveform blob/SQLite access.
- Keep Phase 16 scheduler/backpressure out of scope, but make audio requests carry generation and target timeline microseconds so Phase 16 can schedule them later.

</specifics>

<deferred>
## Deferred Ideas

- Full priority-aware audio scheduling/backpressure belongs to Phase 16.
- Production retiming/speed curves and advanced effects are Phase 18 scope, though Phase 15 must leave typed extension points.
- Mobile/server audio output and cloud render transport are later binding/runtime phases.
- Large Jianying UI alignment remains an ongoing UI workstream; Phase 15 should improve audio playback/waveform production behavior without turning into a full visual redesign phase.

</deferred>
